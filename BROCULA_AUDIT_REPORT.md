## Performance Issues Analysis

### 1. Cache Policy (Server-Side)

**Issue:** Static assets served with 0ms cache TTL  
**Impact:** High - Repeat visits can't leverage browser cache  
**Root Cause:** Python's simple HTTP server (`python3 -m http.server`) doesn't set Cache-Control headers  
**Affected Files:**
- `main-*.js` (3.1 MB)
- `vendor-ui-naive-*.js` (747 KB)
- `vendor-*.js` (698 KB)
- `main-*.css` (121 KB)

**Solution:** Configure cache headers in production (Vercel/nginx):
```
# For hashed assets (1 year cache)
Cache-Control: public, max-age=31536000, immutable

# For HTML (no cache or short cache)
Cache-Control: no-cache
```

### 3. "Unused" CSS/JS (False Positive)

**Issue:** Lighthouse reports 2,018 KiB unused JavaScript and 108 KiB unused CSS  
**Reality:** These are **false positives** for Single Page Applications (SPAs) with lazy loading

**Why:**
- Lighthouse analyzes only the initial page load
- SPA routes are lazy-loaded on demand
- CSS is used by components not rendered on the landing page
- Code splitting intentionally separates rarely-used features

**Evidence:**
- Workspace chunks (e.g., `BasicSystemWorkspace-*.js`, 22KB) are correctly split
- Routes are loaded dynamically as user navigates
- No actual performance impact on user experience
